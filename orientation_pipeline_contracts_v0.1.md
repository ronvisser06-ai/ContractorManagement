# Orientation Generation: Job State and Stage Contracts (v0.1)

Source-of-truth contract for the agent-orchestrated site orientation builder. Every
pipeline stage, the web renderer, and the mobile renderer code against the shapes
defined here. Drop this in `docs/contracts/` and reference it from `CLAUDE.md` so
Claude Code treats it as the spec.

## Conventions

- IDs are prefixed, sortable (ULID under the hood): `job_`, `pkg_`, `mod_`, `blk_`,
  `q_`, `ast_`, `iss_`. Prefixes make logs and QA references readable.
- Large artifacts (extracted deck, content model, quiz, media) live in object
  storage. The job record holds only the storage key plus a `sha256` for each, never
  the payload inline. This keeps the row small and makes every stage cacheable by
  input hash.
- Stages are pure functions of their declared inputs. A stage is idempotent on
  `(job_id, stage, attempt, input_sha256)`: same inputs always reproduce the same
  cache key, so retries and reruns are safe.
- Media is referenced by asset id and resolved to a signed URL only at render time.
  Nothing is base64-inlined in a contract.
- The content model is presentation-agnostic. Web (React) and mobile (Flutter or
  React Native) both interpret the same block types. No HTML is generated by any
  model stage.
- Hyphens only, no em or en dashes, across generated copy and this spec.

## 1. Job state machine

States are owned by the orchestrator (your workflow engine), not by any model. The
QA loop is the only cycle, and it is bounded by `rework_count < max_rework`.

### States

| State | Meaning | Kind |
|---|---|---|
| `queued` | Job created, asset uploaded, not yet started | transient |
| `extracting` | Deterministic parse of PPTX or PDF | working |
| `structuring` | LLM builds the content model | working |
| `generating_quiz` | LLM builds the quiz from the content model | working |
| `qa_review` | Evaluator agent checks coverage, correctness, fidelity | working |
| `awaiting_approval` | Draft ready, a Content Approver must sign off | human gate |
| `publishing` | Freezing and versioning the package | working |
| `published` | Live, assignable to the workforce | terminal |
| `failed` | A stage errored past its retries | terminal (retryable) |
| `cancelled` | Operator or approver cancelled the run | terminal |

### Transitions

| From | Event | To | Side effect |
|---|---|---|---|
| `queued` | `start` | `extracting` | |
| `extracting` | `ok` | `structuring` | write `extracted_deck` ref |
| `extracting` | `error` | `failed` | |
| `structuring` | `ok` | `generating_quiz` | write `content_model` ref |
| `generating_quiz` | `ok` | `qa_review` | write `quiz` ref |
| `qa_review` | `verdict=pass` | `awaiting_approval` | append verdict to `qa_history` |
| `qa_review` | `needs_rework`, `routed_to=structure`, `rework_count < max` | `structuring` | `rework_count += 1` |
| `qa_review` | `needs_rework`, `routed_to=generate_quiz`, `rework_count < max` | `generating_quiz` | `rework_count += 1` |
| `qa_review` | `needs_rework`, `rework_count >= max` | `awaiting_approval` | set `qa_flagged=true`, attach open issues |
| `awaiting_approval` | `approve` | `publishing` | record `approved_by`, `approved_at` |
| `awaiting_approval` | `request_changes`, `target=structure\|generate_quiz` | that state | `rework_count += 1`, attach reviewer notes |
| `awaiting_approval` | `cancel` | `cancelled` | |
| `publishing` | `ok` | `published` | write `OrientationPackage`, set `package_id` |
| `publishing` | `error` | `failed` | |
| any working state | `cancel` | `cancelled` | |
| `failed` | `retry` | re-enter the failed stage | preserve prior artifacts |

Note the deliberate choice: when the QA loop exhausts `max_rework`, the job does not
fail. It advances to `awaiting_approval` with `qa_flagged=true` and the unresolved
issues attached, so a human decides rather than the pipeline dead-ending on safety
content.

## 3. Common stage envelope

Every stage output is wrapped in this envelope. It carries provenance for the audit
trail and the input hashes that make caching and the QA routing deterministic.

```json
{
  "job_id": "job_01J9Z3...",
  "stage": "structure",
  "attempt": 1,
  "schema_version": "0.1",
  "produced_at": "2026-06-22T15:07:40Z",
  "produced_by": {
    "kind": "llm",
    "model": "claude-sonnet-4-6",
    "stage_impl_version": "structure@0.3.1"
  },
  "input_refs": {
    "extracted_deck_sha256": "a1b2c3..."
  },
  "payload": { }
}
```

`produced_by.kind` is one of `code`, `llm`, `agent`, `human`. The `payload` is the
stage-specific contract below.

## 4. Stage contracts

### 4.1 Extract (deterministic) produces `ExtractedDeck`

Pure `python-pptx` or PDF parse. No LLM. PPTX slides and PDF pages normalize into
the same `slides[]` model so downstream stages do not branch on source type.

shared contract for both renderers and for the quiz and QA stages. Every block keeps

### 4.3 Render contract (consume only, no generation)

The renderers are fixed code. The contract is the closed set of block types. Adding a
type means updating both renderers, never asking a model to emit markup.

| Block type | Required fields | Render notes |
|---|---|---|
| `heading` | `level` (1-3), `text` | Section header |
| `paragraph` | `text` | Body copy |
| `list` | `ordered` (bool), `items[]` | Bulleted or numbered |
| `key_point` | `text` | Emphasized takeaway |
| `callout` | `variant` (`info\|warning\|critical`), `title`, `text` | Banded notice |
| `hazard` | `hazard`, `description`, `severity`, `controls[]` | Hazard-control card |
| `image` | `asset_id`, `alt`, `caption?` | Resolves to signed URL |
| `video` | `asset_id`, `caption?`, `poster_asset_id?` | Resolves to signed URL |
| `table` | `headers[]`, `rows[][]`, `caption?` | Responsive table |

Every block also carries `id` and `source_ref`. Unknown block types render as a
visible placeholder in dev and are rejected by schema validation in production.

### 4.4 Generate quiz (LLM) consumes `ContentModel`, produces `Quiz`

Each question cites the source blocks it tests. This citation is the audit artifact
and the input the QA stage and the approver use to verify answers.

```json
{
  "meta": {
    "pass_threshold": 0.8,
    "attempts_allowed": 3,
    "shuffle_questions": true,
    "shuffle_options": true,
    "question_count": 12
  },
  "questions": [
    {
      "id": "q_07",
      "module_id": "mod_03",
      "objective_id": "obj_03_1",
      "source_refs": ["blk_03_02", "blk_03_03"],
      "type": "single_choice",
      "difficulty": "application",
      "stem": "A contractor finds a confined space permit issued on the previous shift. What is required before entry?",
      "options": [
        { "id": "opt_a", "text": "Entry is allowed; the permit is still on file." },
        { "id": "opt_b", "text": "The permit must be re-validated for the current shift." },
        { "id": "opt_c", "text": "A supervisor signature is enough without re-testing." },
        { "id": "opt_d", "text": "Entry is allowed if gas was tested yesterday." }
      ],
      "correct_option_ids": ["opt_b"],
      "rationale": "Permits expire each shift and the atmosphere must be re-tested; supported by blk_03_02 and blk_03_03."
    }
  ],
  "coverage_map": {
    "obj_03_1": ["q_07"]
  }
}
```

`coverage_map` lets the QA stage check that every learning objective (and every
critical hazard via `hazard_index`) is exercised by at least one question without
re-reading the whole quiz.

### 4.5 QA evaluate (agent) consumes deck + content model + quiz, produces `QAVerdict`

The one genuinely agentic stage, run with its own isolated context as a subagent.
`routed_to` is what the orchestrator reads to decide the loop transition.

```json
{
  "verdict": "needs_rework",
  "scores": {
    "coverage":   { "value": 0.91, "pass": true },
    "correctness":{ "value": 0.83, "pass": false },
    "fidelity":   { "value": 0.97, "pass": true }
  },
  "issues": [
    {
      "id": "iss_01",
      "severity": "blocker",
      "category": "correctness",
      "target_stage": "generate_quiz",
      "target_ref": "q_07",
      "description": "Option opt_d is defensible under the source text and could be marked correct by a careful reader.",
      "suggested_fix": "Rewrite opt_d so it is unambiguously incorrect, or cite the specific control that excludes it."
    }
  ],
  "routed_to": "generate_quiz",
  "decision": "rework",
  "rework_count": 1,
  "max_rework": 3
}
```

Rules the orchestrator enforces:

- `verdict` is `pass` only when every score `pass` is true and there are no `blocker`
  or `major` open issues.
- `routed_to` is `structure`, `generate_quiz`, or `none`. When issues span both
  stages, route to `structure` first (the upstream artifact), since a fixed content
  model may resolve downstream quiz issues.
- `decision` is `proceed` (verdict pass), `rework` (within budget), or `escalate`
  (budget exhausted, hand to the approver with issues attached).
- The QA stage never edits artifacts. It only judges and routes.

### 4.6 Publish consumes the approved draft, produces `OrientationPackage`

Freezes the content model and quiz into an immutable, versioned package. Completion
records pin to `content_hash`, so a later edit never silently changes what a
contractor was certified against.

```json
{
  "id": "pkg_kitimat_01_v3",
  "site_id": "site_kitimat_01",
  "version": 3,
  "supersedes": "pkg_kitimat_01_v2",
  "content_model_ref": { "storage_key": "...", "sha256": "..." },
  "quiz_ref": { "storage_key": "...", "sha256": "..." },
  "asset_manifest": [
    { "asset_id": "ast_logo_01", "storage_key": "...", "mime": "image/png" }
  ],
  "content_hash": "sha256:3a7bd3e2...",
  "requalification_policy": "new_content_only",
  "qa_flagged": false,
  "approved_by": "user_jspencer",
  "approved_at": "2026-06-22T16:10:00Z",
  "published_at": "2026-06-22T16:11:20Z"
}
```

`requalification_policy` is the version-bump decision surfaced to the Content Approver
at approval time: `full` (everyone re-does it), `new_content_only` (only
changed modules), or `none` (cosmetic change, no re-qualification).

## 5. Shared TypeScript types

Normative types for the TS side (orchestration, web, mobile). The Python extractor
emits `ExtractedDeck` JSON that conforms to the same shape. Generate these into a
shared package both apps import.

## 6. Orchestration notes

- The workflow engine (Inngest, Trigger.dev, or Temporal) owns the state machine in
  section 1. Each state is one durable step. A failed step resumes from its own
  artifacts, never from the top.
- Caching key per stage: `(job_id, stage, attempt, input_sha256)`. On a QA rework
  loop, only the routed stage and everything after it re-run; upstream artifacts are
  reused by hash.
- The UI subscribes to job status (Supabase realtime) and renders a stage tracker
  plus, at `awaiting_approval`, the side-by-side review screen (each question next to
  the `source_refs` blocks it cites, and any `qa_flagged` open issues).
- Model selection: Sonnet for `structure` and `generate_quiz`, a stronger model for
  the `qa_review` evaluator. Cost is immaterial at this frequency, so bias QA toward
  thoroughness.
- Build order for the skeleton: implement the job record, the state machine, the
  deterministic `extract` step, and the fixed renderer first, with `structure`,
  `generate_quiz`, and `qa_review` stubbed to return canned envelopes that satisfy
  these contracts. That proves orchestration, storage, the realtime tracker, and the
  approval gate end to end before any prompt engineering.

## 7. Approval-gate editing scope (the bounded human editor)

At `awaiting_approval`, the reviewer can correct the AI's output before approving. This
is the only authoring surface in the MVP — the reconciliation of the deferred
drag-and-drop builder (see ExecutionPlan.md §1). Its scope is **deliberately bounded** so
it does not regrow into a free-form WYSIWYG editor. The editor operates directly on the
`ContentModel` and `Quiz` JSON, never on layout or markup.

**Allowed (in scope):**

- Edit the **text** of a block (`heading`, `paragraph`, `list` items, `key_point`,
  `callout`, `hazard` fields, `table` cells, captions, `alt` text).
- **Reorder** blocks within a module and **reorder** modules.
- **Delete** a block or a module.
- **Insert** a block only from the closed block-type set (§4.3), positioned in the flow.
- Edit a **question**: stem, options, the correct answer(s), and rationale.
- **Add or remove** a question; toggle `shuffle_*`; set `pass_threshold` and
  `attempts_allowed`.
- Resolve a `qa_flagged` issue and set the `requalification_policy` at publish.
- Resolve an unresolved media warning (e.g. a `linked_missing` video) by attaching or
  removing the asset.

**Excluded (out of scope for MVP):**

- No free-form canvas, absolute positioning, columns, or custom layout.
- No new block *types*, fonts, colors, or theming (branding comes from the deck/extract).
- No raw HTML/markdown/CSS entry.
- No per-learner branching or conditional logic.
- No drag-and-drop slide/filmstrip authoring (that was the deferred Phase-3 builder).

Every edit preserves each block's `id` and `source_ref` where one exists, and produces a
new stage envelope with `produced_by.kind = "human"` so the audit trail records reviewer
changes distinctly from model output. Edits re-run schema validation (unknown block types
are rejected) before the job can advance to `publishing`.
